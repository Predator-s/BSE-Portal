1 · What BSE Portal is

BSE Portal is an internal web application for a stock-broking firm. It brings together two very different data sources and presents them to staff through role-appropriate screens:

  • The BSE exchange feed — client master data and trade records. In the real world this feed is painful: a full pull takes 5–10 minutes, the network kills any request after 30 seconds, and roughly one pull in five fails partway through. Part A of this project (Mock BSE API) faithfully simulates that behaviour.
  • The firm's internal application — employees and the employee ↔ client mapping (which relationship manager looks after which client). This source is fast and reliable.

From BSE trades combined with the internal mappings, the portal computes relationship-manager incentives (a percentage of the brokerage earned on a manager's mapped clients' trades).

The promise. Every screen loads in under a second — even if BSE is completely down — and open screens update on their own when fresh data arrives. No manual refresh, ever.

2 · How it works — the workflow

The trick behind the promise is a clean split: ingestion and serving never touch each other. A background worker slowly mirrors BSE into our own database; the portal only ever reads that database. So the slow, unreliable feed can never make a screen slow or unavailable.

What happens, step by step

1
Ingest continuously. The sync worker pulls BSE in small pages, each finishing well under the 30-second limit. The instant internal endpoints are pulled every cycle.
2
Survive failures. If a page times out or dies mid-transfer, the worker retries it with backoff and resumes from the last saved position — never from the start, and never committing half a page.
3
Never duplicate. Rows are keyed by their exchange id and upserted, so retries and overlapping refreshes can't create duplicates or contradictions. The rows and the resume position are saved together in one transaction.
4
Serve instantly. Screens read only from Postgres — typically 8–48 ms — with no path to BSE, so they stay fast even if the feed is offline.
5
Update live. The moment new rows are committed, the server pushes a WebSocket event and any open screen refreshes itself.

3 · Getting started

You need Node.js 20+ and Docker (for Postgres). From the project root:

npm install            # install all workspaces
npm run db:up          # start Postgres (Docker)
npm run prisma:migrate # create the database schema
npm run seed           # load roles, features & demo users
npm run dev            # start mock API + portal server + web UI

Once running, open the portal in your browser:

Portal (web)
localhost:5173
Portal API
localhost:4002
Mock BSE API
localhost:4001
Tip. On first launch the worker begins mirroring BSE. Watch the clients and trades counts in the top bar climb — that is the background ingestion filling your local store.

4 · Signing in & roles

The sign-in screen lets you type credentials or pick a ready-made demo account. Every demo account uses the password password. Your account's organisation and role decide which screens you see and how much data each screen shows.

How access is decided

Access is a combination of three things — Organisation × Role × Feature — and each grant carries a scope that narrows the rows you can see:

ScopeMeaningExample
ALLevery rowManagement sees all clients & all incentives
OWNonly the signed-in user's own recordsAn RM sees only their own incentive
MAPPEDonly rows tied to the user's mapped clients"My Clients" and that RM's trades

5 · The screens — a guided tour

Clients — feature: clients · scope ALL

All clients with key details: PAN, city, trading segment, KYC status, their relationship manager, and when the account was opened. The badge on the right shows the server response time — proof the screen met the sub-second requirement.

Trades — feature: trades · filterable by client & date

Every trade with symbol, side, quantity, price, value and brokerage. Filter by a specific client id and/or a date range. Management sees all trades; a relationship manager sees only their mapped clients' trades.

Employees — feature: employees · scope ALL

Everyone on the platform, their title and department, and how many clients each relationship manager carries.

Incentives — feature: incentives · ALL for management, OWN for an RM

Incentive = 15% of the brokerage on each relationship manager's mapped clients' trades. Management sees every RM with totals up top; an RM sees only their own row.

My Clients & the scoped RM view

A relationship manager signs in to a deliberately smaller portal: no firm-wide Clients, Employees, or Access Control in the sidebar. "My Clients" shows only the clients mapped to them.

6 · Access Control — dynamic RBAC

Administrators get an Access Control screen: a live matrix of Organisation × Role × Feature. Nothing here is hardcoded — every screen a user sees is resolved from this matrix at request time, so changing a cell changes that role's access on their very next action.

Each cell is a dropdown; pick a scope (or "none" to revoke). The change is saved immediately.

Try it. Open the portal as an admin in one tab and as an RM in another. Change the RM role's Trades grant to "none", then navigate in the RM's tab — the Trades screen disappears.

7 · Staying fast & live under a failing feed

Always under a second, even if BSE is down

Because screens read only from our own store, they never wait on the exchange. You can stop the Mock BSE API entirely and the portal keeps serving every screen instantly — it simply stops receiving new data until the feed returns. The response-time badge on each screen makes this visible.

Live updates without refreshing

Use the “+ Simulate BSE trades” button in the top bar to inject fresh trades into the mock feed. On the next sync cycle the worker ingests them and pushes a WebSocket event; the Trades and Incentives screens update on their own. The Live indicator and the counts in the top bar reflect the flow.

Correct under failure

  • Timeouts — each pull is abandoned at 30s and retried, never left hanging.
  • Mid-pull failures — a page that dies partway is retried and resumed from the last saved position.
  • Duplicates — rows are keyed by exchange id and upserted, so retries and overlapping pulls are harmless.
  • Overlapping refreshes — cycles never overlap, and incremental pulls use a high-water mark so re-reading the boundary is safe.
Behind the scenes. In the server log you'll see the story play out: ✗ mid-pull failure → ↻ retry → ✓ synced. That is the resilience working as designed, not an error.
